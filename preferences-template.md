# 求职动态规则与偏好配置（模板）

> 本文件是求职 Agent 的唯一动态规则源。主提示词只保留角色、流程和安全边界；评分、搜索、投递、简历优化、记录分层、偏好学习等可变规则均以本文件为准。

## 目标职位
- 在这里填写你想投的岗位
- 按优先级排序
- 例如：解决方案经理、售前支持、产品经理

## 偏好地点
- 目标城市
- 优先区域（如：高新区、主城区等）

## 薪资偏好
- 当前简历期望薪资
- 筛选底线（多少以下不投）
- 加分区间（多少以上优先考虑）

## 加权偏好

### 加分项
- 列出你看重的关键词，例如：
- 五险一金、年终奖金、带薪年假
- 大中型公司、上市公司、外资
- 行业关键词：云计算、大数据、AI、企业服务等
- 岗位关键词：解决方案、客户需求分析、项目管理等

### 降权/风险项
- 列出你不想投的，例如：
- 纯销售、强BD、强电销
- 岗位职责模糊、公司信息不完整
- 要求预付费用、培训贷等异常情况
- 长期高强度加班、通勤过远
- 技术栈不匹配的岗位类型

## 当前评分模型
采用 8 维评分，总分 100，可根据你的价值观调整：

| 维度 | 权重 |
|---|---:|
| 角色匹配 | 20 |
| 地点偏好 | 10 |
| 工作生活平衡 | 15 |
| 薪酬竞争力 | 20 |
| 公司口碑/稳定性 | 10 |
| 成长潜力 | 10 |
| 行业匹配 | 8 |
| 福利完整度 | 7 |

## 等级规则
- A：90-100
- B：80-89
- C：70-79
- D：60-69
- F：60 以下
- A/B 为优先投递候选
- C 级可作为备选或需用户明确确认
- D/F 默认不推荐投递

## 岗位评估模块规则
每个需要详细评估的岗位必须包含 A-G 模块：

| 模块 | 名称 | 内容 |
|---|---|---|
| A | 职位总结 | 岗位、公司、职责重点、适合程度 |
| B | 简历匹配 | 经历匹配度、优势、短板 |
| C | 职级策略 | 投递级别、冲刺可能、谈判点 |
| D | 薪酬调研 | 结合薪资、城市、行业、期望薪资判断是否值得投 |
| E | 个性化建议 | 投递建议、简历强调点、沟通策略 |
| F | 面试准备 STAR+R | Situation、Task、Action、Result、Reflection |
| G | 真实性核查 | 红旗、风险等级、需确认事项 |

真实性核查规则：至少 2 个独立红旗标记高风险；证据不足标记"需确认"，不轻易判定诈骗。

## 岗位分析深度规则
搜索得到的岗位分三层处理：

| 层级 | 适用范围 | 必做内容 | 写入位置 |
|---|---|---|---|
| 原始记录层 | 所有搜索结果 | 保存原始岗位信息，不做详细分析 | job_search_archive.md |
| 批量初评层 | 所有去重岗位 | 评分、等级、主要风险点、主要工作内容、初筛状态 | job.md 摘要；job_detail.md 批量表 |
| 详细评估层 | A/B岗位、用户点名岗位、准备投递岗位、高薪但风险不明岗位、Agent探索潜力岗位 | 评分明细、A-G模块、面试准备、风险核查 | job_detail.md |

执行规则：
- 不要求所有岗位都完整 A-G，但所有去重岗位都必须至少完成批量初评
- 明显不匹配岗位只做简评和归类，不展开 A-G
- 可能达到 A/B、准备投递、用户点名或风险不明的岗位必须补全 A-G
- 不得因为上下文限制跳过批量初评；如果一次处理不完，应分批处理并写入记录
- /投递 前如发现仍有"待初评"岗位，必须先按本规则完成批量初评

## 岗位搜索数量规则
- 每轮岗位搜索目标：至少收集 50 个去重后的候选岗位
- 按"目标职位"顺序视为搜索优先级，越靠前优先级越高
- 当有 4 个待搜岗位方向时，默认配额为：第 1 优先级搜索 20 个；第 2、3、4 优先级各搜索 10 个
- 如果待搜岗位方向不是 4 个，则按以下原则分配：最高优先级至少占 40%，其余方向平均分配剩余名额
- 优先使用分页搜索补足每个方向的配额；如果某方向结果不足，则将剩余配额顺延给下一优先级方向
- 每轮搜索至少保留 1 个"Agent 自动探索岗位"名额，由 Agent 基于简历、市场机会和相邻岗位自动判断关键词。自动探索关键词不得直接使用"数字化转型相关岗位"这类笼统表述，必须转换为具体可搜的关键词，例如：数字化咨询顾问、业务需求分析师、企业数字化、数字产品经理、IT产品经理、数字化项目经理、数字化运营等。Agent 自动探索岗位必须满足：与简历能力可迁移、薪资不明显低于期望、Module G 不高风险
- 去重依据优先使用 Job ID；没有 Job ID 时使用"公司名+岗位名+地点"
- 若平台结果不足 50 个，必须说明实际获取数量和不足原因
- 搜索结果必须按"岗位记录分层规则"保存，不能只在对话中输出
- 每轮搜索应控制频率，避免连续高频请求；分页或多关键词搜索之间保持克制

## 岗位词更新与防过拟合规则
- 每轮投递结束后，必须根据用户最终选择更新目标职位/岗位关键词
- 对用户选择"投"的岗位，提取岗位名、职责关键词、行业关键词，作为后续搜索的正向岗位词候选
- 对用户选择"暂不投"或"pass"的岗位，若用户说明原因，则提取负向岗位词或降权因素；若未说明原因，只记录状态，不做过度推断
- 更新目标职位时不得完全被用户本轮选择绑定，避免过拟合
- 每轮搜索至少保留 1 个"Agent 自动探索岗位"名额
- Agent 自动探索岗位需要在 job.md 中标记来源：Agent探索
- 自动探索岗位的关键词不得使用"数字化转型相关岗位"这类笼统表述，必须转换为具体可搜关键词

## 投递与简历优化规则
- 未经用户明确确认，不得自动投递岗位
- 当用户确认投递某岗位时，询问是否对该岗位生成优化简历
- 若用户选择"是"，则生成优化简历 PDF 文件
- 若用户选择"否"，则直接投递猎聘在线原始简历
- 优化简历必须基于猎聘接口获取的简历内容进行优化，生成优化简历前先调用猎聘 MCP 获取最新简历。
- 优化简历必须保留项目经历，可针对岗位职责调整凸显方向和描述侧重点，但不得删除任何模块、条目或经历，不得大幅精简内容。
- 优化原则：
  - 不编造经历、不夸大职级。实际参与程度、负责范围、职级头衔必须如实反映，不得为匹配 JD 而拔高。
  - 优先重组表达方式和调整凸显方向，使之与 JD 匹配，但不删减原有内容。
  - 保留简历全部模块和条目，不得因为"只和 JD 无关"而删除任何经历。
  - 可以通过调整描述侧重点、前置相关项目、补充 JD 关键词的方式来匹配，而不是精简。
  - 履历中的工作量、复杂度、贡献描述要如实反映，不得往 JD 上靠而夸大实际参与程度。
  - ATS 友好：保持标准段落格式，合理使用 JD 中的关键词，但不堆砌无关词汇。
- PDF 文件统一保存到项目根目录下的"优化简历"目录
- 文件命名格式：公司名+岗位名.pdf
- 文件名需清理 Windows 非法字符：\\ / : * ? " < > |
- 生成 PDF 后，必须向用户展示 PDF 路径
- 如果当前 MCP/招聘平台接口不支持指定 PDF 附件投递，不得假装已发送 PDF；应明确告知用户，并记录为"PDF 已生成但未随 MCP 投递发送"
- 若需要修改猎聘在线简历，必须先展示修改内容并等待用户确认

### 简历优化顾问工作流
当用户要求"优化简历"或"生成优化简历"时，按以下步骤执行：

**第一步：JD关键词分析**
- 先提炼JD的核心关键词、能力要求和筛选重点
- 输出：JD关键词分析（岗位核心能力、硬性要求、加分项、筛选关键词）

**第二步：简历优化建议**
- 对比用户猎聘简历与JD的匹配度
- 指出优势、差距、需补充的方向
- 输出：简历优化建议（匹配度评分、优势项、待补充项、优化方向）

**第三步：优化简历内容**
- 基于猎聘接口获取的最新简历内容进行优化
- 不编造经历、不虚构数据，缺少数据用【待补充数据】标出
- 语言专业、简洁、有结果感，避免空话和AI味
- 经历表述使用"动作+方法+结果+能力体现"的句式
- 保留简历全部模块和条目，不得因为"只和JD无关"而删除任何经历
- 可以通过调整描述侧重点、前置相关项目、补充JD关键词的方式来匹配

**第四步：生成HTML+CSS版式代码**
- 精英白领职场风格A4简历成品
- 顶部通栏蓝色标签"个人简历 RESUME"（用 grid 布局，禁用 position: fixed 避免打印白块）
- 浅灰/白底，搭配蓝色点缀，干净高级
- 姓名区：居中显示，最大宽度对齐左侧栏（72mm），不撑满
  - 中文名大字18pt，英文名/拼音大写9pt
  - 下方小字：解决方案经理 / AI产品运营方向 | 15年+工作经验 | 西安 | 求职意向
- 左侧栏：浅灰色背景(#f8f9fc)，宽度72mm
  - 照片区：55mm宽x72mm高，圆角，嵌入 base64 图片（不依赖本地文件）
  - 基本信息区
  - 求职意向区
  - SKILLS技能区：技能名+横向进度条（蓝色渐变）
- 右侧内容区：工作经验/项目经验，每段按"公司/项目名+职务+时间+3-5条成果"呈现
- 底部通栏"意向"横条：蓝色渐变背景，展示目标岗位/行业/薪资
- 整体不要花哨，不要像普通WORD文档，要像排好版的正式视觉简历
- 页面尺寸为A4大小，grid布局：grid-template-columns: 72mm 1fr; grid-template-rows: auto auto 1fr auto
- 一个HTML文件即可运行，包含完整HTML/CSS
- 网页里的文字可在线修改：加 [contenteditable="true"] 到 .page 容器
- CSS必须包含强制打印背景：* { -webkit-print-color-adjust: exact; print-color-adjust: exact; }

**第五步：保存并导出PDF指引**
- HTML文件保存到"优化简历"目录，命名格式：公司名+岗位名.html
- 向用户说明导出PDF步骤：
  1. 用 Edge/Chrome 打开 HTML 文件
  2. 按 Ctrl+P 调出打印对话框
  3. 展开「更多设置」，勾选 ☑️ **打印背景图形**（关键！否则蓝色条和技能条会消失）
  4. 目标打印机选「另存为PDF」
  5. 保存即可
- 向用户说明：文字可直接在网页上点击修改，改完再重新打印PDF

## /投递 命令规则
当用户输入"/投递"时，执行以下流程：
1. 先读取 preferences.md 和 job.md 主索引
2. 检查 job.md 中是否存在"待初评"岗位
3. 如果存在"待初评"岗位，必须先按当前评分模型进行批量初评：
   - 对明显不匹配岗位可简评，但必须给出评分、等级、主要风险点和主要工作内容
   - 对可能达到 A/B 的岗位必须补全评分明细和模块A-G
   - 初评结果按"岗位记录分层规则"写回对应文件，不能只在对话中输出
4. 从全部已评测岗位中筛选候选岗位：
   - 评分等级为 A 或 B
   - Module G 不为高风险
   - 用户选择不是"pass"或"暂不投"
   - 未记录为已投递
5. 按评分从高到低列出所有符合条件的岗位，展示序号、岗位名称、公司、评分、薪资、地点、主要风险点、主要工作内容
6. 询问用户选择序号，支持单选、多选或"全部"
7. 用户选择后，逐个执行：展示岗位名称、公司名称、薪资、地点，询问是否生成优化简历，再次确认是否投递
8. 用户最终确认后，才调用猎聘 MCP 投递
9. 投递完成后按"岗位记录分层规则"更新记录，并按"岗位词更新与防过拟合规则"更新偏好
10. 如果完成批量初评后仍没有符合条件的岗位，提示需要先搜索更多岗位或调整筛选规则

## /捞回 命令规则
当用户输入"/捞回""捞回不投岗位""复盘不投岗位"或类似意图时，执行以下流程：
1. 先读取 preferences.md 和 job.md 主索引。
2. 从 job.md 中筛选用户选择为"暂不投"、"pass"、"未归类"，且未记录为已投递的岗位。
3. 如需确认历史原因、评分明细或风险点，再读取 job_detail.md；默认不读取 job_search_archive.md 和 job_apply_log.md。
4. 按当前 preferences.md 的最新偏好重新初筛，优先展示以下岗位：
   - 原评分为 A/B 或接近 B 的岗位
   - 角色与 AI产品经理、AI交付、解决方案、售前方案、数字化转型相关的岗位
   - 之前暂不投原因较弱、可通过沟通确认的岗位
   - 非纯 Java / Python / Go 开发、非强销售、非高风险岗位
5. 展示时必须包含：序号、岗位名称、公司、原评分、原状态、原不投/pass原因、当前可重新考虑理由、主要风险点、建议动作。
6. 建议动作仅允许为：恢复待定、保持暂不投、保持pass、需要补充核查。
7. 不得直接投递暂不投或pass岗位；只有用户明确选择"恢复待定"或"改为投递候选"后，才可更新 job.md 状态并进入 /投递 候选流程。
8. 用户确认恢复后，按"偏好学习规则"记录本次反馈；不得过度推断用户未说明的原因。

## 岗位记录分层规则

| 文件 | 用途 | 保存内容 |
|---|---|---|
| job.md | 主索引 | 最新状态、评分摘要、投递状态、PDF路径、详情链接；不保存大段原始结果和完整A-G评估 |
| job_detail.md | 岗位详情 | 评分明细、维度权重/得分/原因、A-G模块、面试准备 STAR+R、风险核查、重评记录 |
| job_search_archive.md | 搜索归档 | 搜索条件、批次、关键词、配额、原始结果、去重记录、实际数量和不足原因 |
| job_apply_log.md | 投递日志 | 投递时间、岗位/公司、Job ID/Type、投递结果、PDF路径、是否同步在线简历、打招呼语、最终确认记录 |

读取规则：日常筛选和 /投递 优先读取 job.md；仅在复盘、重评、查看详情、风险核查或确认历史投递时读取其他文件。

更新规则：岗位已存在时不重复新增；job.md 只更新状态和摘要；job_detail.md 更新详细评估；job_search_archive.md 追加搜索批次；job_apply_log.md 追加投递记录。

## 偏好学习规则
- 用户选择"投"的岗位：更新 job.md 主索引；追加 job_apply_log.md；提取行业、岗位名、薪资区间、公司规模、地点、职责关键词、福利关键词、公司阶段作为正向偏好
- 用户选择"暂不投"或"pass"的岗位：更新 job.md 主索引；如果用户说明原因，写入负向偏好或降权因素；如果未说明原因，只记录状态，不做过度推断
- 用户修改评分模型、权重、搜索规则、投递规则、简历优化规则或风险过滤规则：立即更新本文件；后续岗位按新规则重评或筛选；重评摘要写入 job.md，明细写入 job_detail.md
